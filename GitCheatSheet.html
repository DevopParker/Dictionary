Git Cheat Sheet

Quick Setup

Get started by creating a new file or uploading an existing file. We recommend every repository include a README, LICENSE, and .gitignore.

Create a new repository on the command line:

echo "# MOOCJava" >> README.md
git init
git add README.md
git commit -m "first commit"
git branch -M main
git remote add origin https://github.com/Username/Repository.git
git push -u origin main

What's happening:

  • echo "# MOOCJava" >> README.md - Creates a README file with a header
  • git init - Initializes a new Git repository in the current directory
  • git add README.md - Stages the README file for commit
  • git commit -m "first commit" - Creates your first commit with the staged file
  • git branch -M main - Renames the default branch to "main"
  • git remote add origin [URL] - Links your local repository to the remote GitHub repository
  • git push -u origin main - Pushes your code to GitHub and sets up tracking

Push an existing repository from the command line:

git remote add origin https://github.com/Username/Repository.git
git branch -M main
git push -u origin main

What's happening:

  • git remote add origin [URL] - Connects your existing repository to GitHub
  • git branch -M main - Ensures your primary branch is named "main"
  • git push -u origin main - Uploads your code and sets the upstream reference

Recommended Git Workflow

Step-by-Step Development Workflow

Step 1: Set up your environment

git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global color.ui auto

Step 2: Create or clone a repository

# Option 1: Create a new repository
git init project-name
cd project-name

# Option 2: Clone an existing repository
git clone https://github.com/username/repository.git
cd repository

Step 3: Create a feature branch

git checkout main     # Start from main branch
git pull             # Make sure you're up to date
git checkout -b feature/my-feature

Always work in a feature branch, never directly on main.

Step 4: Make changes and commit regularly

# Check status often
git status

# Add specific files or all changes
git add file.txt
git add .

# Commit with descriptive message
git commit -m "Add login functionality"

Make focused, logical commits with clear messages.

Step 5: Stay in sync with main

git checkout main
git pull
git checkout feature/my-feature
git merge main

Regularly incorporate changes from main to avoid large merge conflicts later.

Step 6: Push your branch

git push -u origin feature/my-feature

The -u flag sets up tracking so you can just use "git push" next time.

Step 7: Create a Pull Request

Go to your repository on GitHub or similar platform:

  • Click "Compare & pull request"
  • Add a title and detailed description
  • Request reviews from team members

Step 8: Address feedback

# Make changes based on reviews
git add .
git commit -m "Address review feedback"
git push

The pull request will update automatically.

Step 9: Merge and clean up

# After PR is approved and merged
git checkout main
git pull
git branch -d feature/my-feature       # Delete local branch
git push origin --delete feature/my-feature   # Optional: delete remote branch

Essential Git Commands

Git Configuration

git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global color.ui auto

Configuration is the first step in any Git workflow. These settings identify you as the author of your work and customize your Git environment.

Starting a Project

git init [project name]      # Create a new local repository
git clone <project url>     # Download a project with entire history

You can either start a new project from scratch with init or download an existing project with clone.

Day-to-Day Work

git status                  # Display current working directory status
git add [file]               # Add file to staging area
git diff [file]              # Show changes between working directory and staging
git diff --staged [file]     # Show changes between staging and repository
git checkout -- [file]       # Discard changes in working directory
git reset [path]             # Unstage changes
git commit                   # Commit staged changes
git commit -m "message"      # Commit with inline message
git rm [file]                # Remove file from working directory and staging

Working with Branches

git branch                  # List local branches
git branch -a               # List all branches (local and remote)
git branch [branch_name]    # Create new branch at current HEAD
git checkout [branch_name]  # Switch to specified branch
git checkout -b [branch]    # Create and switch to new branch
git merge [branch_name]     # Merge branch into current branch
git branch -d [branch_name] # Delete a branch
git rebase [branch_name]    # Rebase current branch onto specified branch

Branches are Git's way of enabling parallel development. Create feature branches for new work and merge them back when ready.

Stashing Changes

git stash                  # Save current changes temporarily
git stash pop               # Apply stashed changes and remove from stash
git stash drop              # Delete a stash

Stashing allows you to save your work-in-progress when you need to switch contexts or branches without committing incomplete work.

History and Inspection

git log                             # View commit history
git log --oneline --graph --decorate   # Compact history with branch visualization
git reflog                           # List operations made on local repository

Tagging

git tag                    # List all tags
git tag [name] [commit]     # Create tag for specified commit
git tag -a [name] -m "msg"  # Create annotated tag
git tag -d [name]           # Delete a tag

Tags are references to specific points in Git history and are typically used to mark release points (v1.0, v2.0, etc.).

Undoing Changes

git reset [--hard] [target]  # Reset current HEAD to specified state
git revert [commit]         # Create new commit that undoes changes

Caution: git reset --hard will discard all uncommitted changes. Make sure you want to do this before running the command.

Remote Operations

git fetch [remote]          # Download objects and refs from remote
git pull [remote]           # Fetch and merge from remote
git push [--tags] [remote]  # Push local changes to remote
git push -u [remote] [branch] # Push and set upstream reference

Cloning

Clone a repository:

git clone https://github.com/Username/Repository.git

Downloads a copy of the repository to your local machine with full history.

Clone a specific branch:

git clone -b branch-name https://github.com/Username/Repository.git

Clones the repository but checks out the specified branch instead of the default branch.

Clone with a specific directory name:

git clone https://github.com/Username/Repository.git custom-directory-name

Clones the repository into a folder with the specified name instead of the default repository name.

Shallow clone (only recent history):

git clone --depth=1 https://github.com/Username/Repository.git

Creates a shallow clone with only the most recent commit, useful for large repositories when you don't need the full history.

Forking

Fork workflow:

  1. Fork the repository on GitHub (click the "Fork" button)
  2. Clone your fork locally: 
    git clone https://github.com/YOUR-USERNAME/Repository.git
  3. Add the original repository as upstream: 
    git remote add upstream https://github.com/ORIGINAL-OWNER/Repository.git
  4. Verify remotes: 
    git remote -v

What is forking? Forking creates a personal copy of someone else's repository on GitHub. This lets you freely experiment with changes without affecting the original project. Forks are commonly used to propose changes to someone else's project or to use someone else's project as a starting point for your own idea.

When to use forking: Use forking when you want to contribute to a project you don't have write access to, or when you want to use someone else's project as a base for your own work.

Sync your fork with the original repository:

git fetch upstream
git checkout main
git merge upstream/main
git push origin main

This pulls in changes from the original repository to keep your fork up to date.

Ignoring Files

cat < .gitignore
/logs/*
!logs/.gitkeep
/tmp
*.swp
EOF

To ignore files, create a .gitignore file in your repository with a line for each pattern. File ignoring will work for the current and sub directories where .gitignore file is placed. In this example, all files are ignored in the logs directory (excluding the .gitkeep file), whole tmp directory and all files *.swp.

Understanding Git Terminology

  • Repository: The collection of files and their complete history
  • Working directory: The files that you see and edit
  • Staging area: A place to prepare what goes into your next commit
  • Commit: A snapshot of your files at a specific point in time
  • Branch: A parallel version of a repository
  • Remote: A version of your repository hosted elsewhere (like GitHub)
  • Pull Request: A way to propose changes and request review
  • Merge: Combining changes from different branches

Note: Always be careful with commands that can destroy history, like git reset --hard or git push -f. These can permanently delete work if used incorrectly.
